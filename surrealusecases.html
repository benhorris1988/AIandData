Graph-Based Data Provisioning

Security & Access Control

The Challenge in SQL

Row-level, column-level, table, and schema security is notoriously tricky to manage at scale in standard SQL Server. It often requires complex dynamic SQL, nested views, and triggers that degrade query performance and are hard to audit.

The SurrealDB Test

We will model our data provisioning natively as a Graph. Instead of static mapping tables, we'll create edge relationships representing 'access' directly linking User/Group Node -> HAS_ACCESS_TO -> Dataset/Schema Node. Furthermore, we will test SurrealDB's DEFINE PERMISSIONS functionality to enforce this graph logic at the database layer securely.

-- Example of Graph-based Row Level Security in SurrealQL
DEFINE TABLE patient_records SCHEMAFULL
  PERMISSIONS
    FOR select, update WHERE $auth.id ->manages->id;

Target Outcome

Prove that we can completely out-maneuver traditional relational data security by visualizing and enforcing access rights dynamically via Graph relationships. This would massively simplify our API and provisioning application architecture.

Staging & Transformation Workloads

Pipeline Modernization

The Scenario

Moving heavily transformed staging data out of our existing SQL Server implementation and recreating those workflows natively.

The SurrealDB Test

Trial out the transformation workloads we currently orchestrate in SSIS and SQL Server Stored Procedures using SurrealQL's scripting and functional capabilities. We will construct event-driven pipelines utilizing SurrealDB events to reshape data upon insertion.

Target Outcome

Evaluate if the functional nature of SurrealQL provides a sufficient, cleaner alternative to T-SQL for ELT tasks, potentially allowing us to phase out legacy SSIS packages entirely for specific data domains.

The GOLD Layer Evolution

AI & Semantic Search

The Scenario

Taking our curated, relational GOLD layer data and upgrading it to support next-generation Agentic AI and retrieval workloads.

The SurrealDB Test

We will migrate a subset of the GOLD layer, physically structure it into a Graph, and generate semantic Vector Embeddings for unstructured text columns (like notes or histories), storing them natively in the same records.

Target Outcome

Demonstrate the power of a "Hybrid Search" query—finding records based on semantic vector similarity, and then immediately traversing its Graph connections, all within a single SurrealQL statement for our AI integrations.

Graph Performance Benchmarking

Scale & Viability

The Scenario

Stress-testing deep entity relationships (e.g., recursive hierarchal data, supply chain mapping, or identity tracking).

The SurrealDB Test

Run a parallel ingestion and traversal test. We will build identical deep-link queries in SurrealQL, Microsoft Fabric Graph (using standard SQL graph extensions), and Kusto Query Language (KQL) to evaluate raw compute efficiency.

Target Outcome

Establish hard baseline metrics for Query Latency, Ingestion Speed, and Developer Syntax Complexity, objectively validating if SurrealDB outperforms the Microsoft ecosystem in native graph operations.

Schema-less Telemetry Logging

Document Store Capabilities

The Scenario

Capturing un-schematized application logs, IoT streams, or volatile 3rd-party API responses.

The SurrealDB Test

Bypass rigid SQL table definitions entirely. Ingest volatile JSON payloads directly into SurrealDB document tables and leverage dynamic indexing to query across unpredictable attributes.

Target Outcome

Measure ingestion latency and query performance against SQL Server's OPENJSON functionally, determining if SurrealDB makes upstream data engineering pipelines more resilient to schema changes.

Real-time "Live" Dashboards

Event Streaming

The Scenario

Powering "command center" style operational BI dashboards that require sub-second updates, traditionally requiring polling or SignalR overhead.

The SurrealDB Test

Deploy a frontend client utilizing a SurrealDB LIVE SELECT connection. As background systems ingest or alter records in the database, verifying the push notifications directly to the browser via WebSockets.

Target Outcome

Prove that we can eliminate our custom event-bus middle tier and rely entirely on the operational database to act as a resilient pub-sub message broker for UI updates.
